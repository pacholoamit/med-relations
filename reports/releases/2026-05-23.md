## Summary

Fixes token parsing in `AccountLoader` so URL-encoded `token` query params are decoded correctly. Previously, parsing relied on `URLSearchParams`, which decodes `+` to a space — corrupting any token that legitimately contains `+` (e.g., base64-encoded values). This change extracts the raw `token` value from the query string and decodes it with `decodeURIComponent`, so `+` and other percent-encoded symbols round-trip cleanly.

## Changes

- `src/admin/core/AccountLoader.js`
  - After the existing `URLSearchParams` pass, walk the raw query string (`urlObj.search`) and locate the `token` pair manually.
  - Decode the raw `token` value with `decodeURIComponent`, falling back to the raw value if decoding throws.
  - Overwrites the `token` entry in the parsed result so downstream code receives the intended value with `+` preserved.

### [microservices] fix(statara): demograph activate matcher + 6h polling + repair hook    (#2127)

**Repo:** [MediaJel/amplication-nestjs-microservices](https://github.com/MediaJel/amplication-nestjs-microservices)
**Merged:** 2026-05-06T15:42:23Z
**PR:** [#2127](https://github.com/MediaJel/amplication-nestjs-microservices/pull/2127)

## Summary                                                                                                                                 
                                                         
  Two latent defects in `apps/internal-service/src/features/audiences/sources/statara/queues/statara-activate.queue.ts` were causing         
  `audience.statara.activate` to fail with `Error waiting for audience file: {}` and silently corrupting partitions on every "successful"
  run.                                                                                                                                       
                                                         
  - **Substring matcher** (`findActivatedAudienceFile:154-156`, `findAllActivatedAudienceFiles:199`) — `Key.includes(partitionBaseName)`     
  resolved the wrong file for any single-digit partition. S3 keys sort lex-first and `'0' (0x30) < '_' (0x5F)`, so for
  `<id>-statara-part1.csv` the lookup returned `…-part10_smai_append…` before `…-part1_smai_append…`. `part1` silently received `part10`'s   
  data, `part2 → part20`, `part3 → part30`, etc. The wrong key was then persisted to `AudiencePartition.sourcePath` and `isActivated=true`,
  so retries SKIPped the corrupted rows instead of healing them. **Anchor the match on the underscore Statara always inserts**
  (`partitionBaseName + "_"`) so `partN` only matches `partN_…`, never `partN0…partN9`.
  - **30-minute polling timeout** (`processPartition:248`) — Statara emits later partitions in waves spaced 30+ min apart on 50+ partition
  runs (observed gaps of ~95 min between waves on `a38db7ff-…`). `periodicallyCheck` defaulted to `maxAttempts: 360 × 5s = 30min`, so        
  partitions whose outgoing file landed in a later wave timed out before Statara delivered them. **Pass an explicit ~6h budget**
  (`Math.ceil(21_600_000 / pollInterval)`) mirroring the publish-side timeout at `batchPublishPartitions:520`. Also surface `partition.name` 
  in the rejection so the next failure isn't a bare `{}`.

  ### Repair for already-corrupted data                                                                                                      
   
  - `apps/internal-service/scripts/repair-statara-partitions.ts` — scans `AudiencePartition` for rows whose `sourcePath` references a        
  wrong-numbered outgoing key (`…-partM_smai_append…` where `M ≠ N` from the partition's name), resets `isActivated` / `activatedPath` /
  `sourcePath`, and clears the parent `Audience.isActivated` so the next activate cycle re-pulls the correct file from Statara. Idempotent — 
  re-runs find nothing and exit immediately. Supports `--audience-id <id>` (scoped) or `--all` (every Statara audience), plus `--dry-run`.
  - `apps/internal-service/chart/templates/repair-statara-partitions-job.yaml` — Helm `post-install,post-upgrade` Job that runs the script
  automatically after each release. Uses `bun /app/scripts/repair-statara-partitions.ts --all` since the prod image is `oven/bun:1.2-slim`   
  and bun runs TypeScript natively (no ts-node needed). Hook weight `1` so it fires after the new Deployment converges. Gated by
  `Values.repairStataraPartitions` (default `true`); flip to `false` in `chart/stages/<stage>.yaml` to remove the no-op Job from future      
  releases.                                              

  ## Test plan

  - [ ] Activate a new multi-partition Demograph audience (≥10 parts). Spot-check 3+ destination files in MediaJel S3 — content of           
  `…/partN.csv` matches the source for that exact `partN`, never `partN0`.
  - [ ] Confirm `audience.statara.activate` no longer fails at the 30-min mark for audiences whose later partitions land in a delayed Statara
   wave.                                                                                                                                     
  - [ ] Verify on a fresh release with no corruption: Helm hook Job runs, logs `No corrupted partitions found.`, exits 0 in <5 s.
  - [ ] Verify on `a38db7ff-f773-419b-beae-2ecbd7f9f94e`: Job detects partitions 1–9 (or whichever were single-digit-numbered) as mismatches,
   resets them, then a re-triggered activate repopulates with correct data and `Audience.isActivated` flips back to `true`.

### [microservices] fix(statara): require save completed before activate; drop ghost jobs (#2130)

**Repo:** [MediaJel/amplication-nestjs-microservices](https://github.com/MediaJel/amplication-nestjs-microservices)
**Merged:** 2026-05-06T20:03:25Z
**PR:** [#2130](https://github.com/MediaJel/amplication-nestjs-microservices/pull/2130)

Fixes "Activate ✓ but Publish ✗" on Demograph audiences when the save queue is backed up. Server now refuses `/activate` if save isn't completed yet, auto-recovers audiences corrupted by
   the prior bug, and fails loudly at the queue layer if anything bypasses the gate.                                                                                                        
                                                                                                                                                                                            
  ## Root cause                                          

  Activate is enqueued ~6 s after save (`service.activate` uses `delay: 5000`), but save can sit in the queue for tens of minutes when other jobs are ahead of it. When activate ran first, 
  it found `Audience.partitions = []` (save hadn't streamed them in yet), hit the silent-success branch in `statara-activate.queue.ts`:
                                                                                                                                                                                            
  ```ts                                                  
  if (!audience.partitions.length) {
    job.log("No partitions to process");
    return { paths: [] };
  }
  ```

  …and Bull marked the job `completed`. The transfer loop that copies partitions from MediaJel S3 → `0018c00002RaoUpAAJ/incoming/smai_append_via_sii_statara_id_pipeline/` was skipped      
  entirely. `Audience.isActivated` stayed `false`, so the subsequent publish hit `service.publish` (`if (!audience.isActivated) throw …`) and failed with "Failed to publish audience".
                                                                                                                                                                                            
  **Concrete evidence from `4938056f-a4b8-413b-9824-f46e3ecafeb1` (TwentyOnePlusNew 7May2026):**                                                                                            
   
  | Job | Added | Process started | Process finished | Notes |                                                                                                                              
  |---|---|---|---|---|                                  
  | `audience.statara.save` | 12:10:45 AM | **12:36:27 AM** | 12:51:47 AM | 26 min wait, 53 partitions written |                                                                            
  | `audience.statara.activate` | 12:10:51 AM | 12:10:56 AM | **12:10:56 AM** | 11 ms, "No partitions to process" |                                                                         
                                                                                                                                                                                            
  Activate finished 25 min 31 s **before** save started. Nothing was ever uploaded to Statara's incoming folder, so Statara never produced outgoing files, so publish had nothing to        
  consume.                                                                                                                                                                                  
                                                                                                                                                                                            
  ## Fix — three defensive layers                        

  - **`service.activate` precondition** (`statara-audience.service.ts:223-247, :294`) — new `assertSaveCompleted` rejects the request when the save Bull job exists in any state other than 
  `completed`. Error: `Cannot activate audience X: save job is <state>. Wait for save to finish before activating.` If the save job has aged out of Redis, partitions in DB act as the
  proof.                                                                                                                                                                                    
  - **Ghost-completion auto-cleanup** (`statara-audience.service.ts:273-292`) — `checkOngoingActivations` now detects the silent-success signature (`state === "completed" &&
  returnvalue.paths.length === 0`) and removes the stale job so a fresh activate can run. Audiences affected by the prior bug recover with a single click in the UI; no Redis surgery       
  needed.
  - **Queue-layer fail-loud** (`statara-activate.queue.ts:639-646`) — zero-partition branch now throws `Audience X has no partitions; save must complete before activation.` instead of     
  returning success. Any future bypass of the service gate fails visibly in the Bull dashboard.                                                                                             
   
  ## Test plan                                                                                                                                                                              
                                                         
  - [ ] **Recovery of `4938056f-…`**: click Activate from the UI. Service detects the ghost activate job, drops it, `assertSaveCompleted` passes (save is `completed`), enqueues a fresh

  Then publish succeeds.                                                                                                                                                                    
  - [ ] **Premature activate is rejected**: trigger save on a fresh audience, then call `/activate` while save is still `waiting`/`active`. API returns 400 with `Cannot activate audience
  X: save job is <state>. …`.

  - [ ] **Bull dashboard**: post-deploy, no new "0% / No partitions to process" completed activate jobs appear.
  - [ ] **Other Statara variants unaffected**: existing single-partition audiences and email/phone Demograph audiences activate and publish normally (touched logic is shared but the

### [gql-service] [HOTFIX] Scope Salesforce sync attachment lookup by campaignOrder to stop sibling theft (#7420)

**Repo:** [MediaJel/mediajel-gql-service](https://github.com/MediaJel/mediajel-gql-service)
**Merged:** 2026-05-08T01:09:35Z
**PR:** [#7420](https://github.com/MediaJel/mediajel-gql-service/pull/7420)

## Summary

Hotfix for the Salesforce → PandaDoc sync: when an opportunity's proposal number matched **multiple** campaign orders (via the `OR` query in `findDisplayCampaignOrder` over `name_contains`, `IONumber`, and `additionalIONumbers_some`), processing each order would steal the same `Attachment` row from its siblings. The user-visible symptom was the log line `Attachment already exists, updating: <id>` firing on a campaign order whose `attachments` array was nevertheless empty.

## Root cause

`PandadocAttachmentsManager.prepareAttachment` looked up existing attachments by `referenceId` alone — globally, with no campaign-order scope:

```ts
// before
const existingAttachment = await prismaServer.query.attachments({
  where: { referenceId },
});
```

`Attachment.campaignOrder` is a single-valued relation. Each iteration of the `for (const campaignOrder of campaignOrders)` loop in `display-processor.ts` would find that "existing" row (created by a previous iteration or a previous run), then run `updateAttachment` with `campaignOrder: { connect: { id: campaignOrderId } }` — which moves the row off whichever order it used to belong to. The last order processed wins; every earlier order ends up with no attachment.

## Fix

Scope the lookup to the `(referenceId, campaignOrderId)` pair so each order finds **its own** prior row, or none — never a sibling's:

```ts
// after
const existingAttachment = await prismaServer.query.attachments({
  where: {
    referenceId,
    campaignOrder: { id: campaignOrderId },
  },
});
```

Behavior table:

| Scenario | Before | After |
|---|---|---|
| First sync, order A | Creates row on A | Same |
| Sync also matches order B | Moves A's row to B → A loses it | A and B each get their own row |
| Re-sync of order A | Could find B's row, steal it back | Finds A's row, updates in place |

## Files changed

- `src/jobs/salesforce-sync/lib/attachments-manager.ts` — adds `campaignOrderId` to `UpsertAttachmentInput` and scopes the query
- `src/jobs/salesforce-sync/processor/display-processor.ts` — passes `campaignOrderId` through; replaced the stale `// TODO: Make referenceId a unique field` comment (that direction would now be wrong, since multiple orders legitimately share a PandaDoc ID)
- `src/jobs/salesforce-sync/processor/display-processor.test.ts` — new test cases that lock in the fix

## What this does NOT fix

- **Existing rotten data**: rows already moved between orders by the old behavior won't auto-heal — they stay on whichever order they last landed on. A separate cleanup pass is needed if that matters.
- The pre-existing `OR`-based campaign-order match in `findDisplayCampaignOrder` still returns multiple orders for the same proposal; this fix just makes that fan-out safe.

- [ ] `handleUpsertAttachment` — UPDATE path: when an attachment for THIS `(referenceId, campaignOrder)` exists, it's updated in place
- [ ] `handleUpsertAttachment` — SIBLING ISOLATION: when an attachment with the same `referenceId` exists on a DIFFERENT `campaignOrder`, the scoped query returns `[]`, CREATE path runs for this order, the sibling row is not touched

### [gql-service] FIX Publisher transaction & impression counts undercounted in consolidate-report-s3 (#7397)

**Repo:** [MediaJel/mediajel-gql-service](https://github.com/MediaJel/mediajel-gql-service)
**Merged:** 2026-05-11T14:05:35Z
**PR:** [#7397](https://github.com/MediaJel/mediajel-gql-service/pull/7397)

# Description
## Ticket Issue #7368 :  
## Fix Publisher Table Undercount in Consolidate Report
### Problem

The Publishers table on **Campaign Details > Transactions** undercounts impressions and transactions across most campaigns.

`saveTransactionsPublishers` in `src/jobs/consolidate-report-s3/queries/transactions.query.ts` does an `upsert` whose `update` branch **overwrites** scalars with the latest slice instead of accumulating across calls. Per the bug report:

> "Manually invoking `saveTransactionsPublishers` twice in succession against the same reportId with two different non-empty publishers[] slices for the same publisher key... will equal the second call's values, not the sum of both calls."

This affects every campaign reprocessed more than once (i.e. virtually all of them) and every campaign whose S3 transactions report has multiple chunks.

## Solution

Cumulative merge + wipe-then-insert in `processTransactionReport`:

1. **Read prior** persisted publisher rollup from Prisma (added to existing `previousReport` query).
2. **Read new** slice from the freshly-downloaded S3 transactions report.
3. **Merge** via `mergePublishers(prior, new)` — date-deduplicated, **NEW wins on overlap**, PRIOR retained for dates not in NEW. Top-level scalars are recomputed from the merged `byDate` (single source of truth). Same rule applied to nested `byCampaign[*].byDate`.
4. **Wipe** all persisted publisher rows for this `reportId` (parent + nested), then **insert** the merged canonical result.

Date keys are normalized to `YYYY-MM-DD` (Prisma returns `DateTime` ISO strings, S3 uses date-only strings — without normalization the dedupe Map keys never collide).

The merge is **idempotent**: running consolidate against the same input twice produces the same persisted state.

## Test Case (Demo Script)
Reproduces the exact failure mode from the bug report — two `saveTransactionsPublishers` calls on the same `reportId` with different date slices for the same publisher.

**Run:**


- Slice 2: Yahoo on Apr 02, `imp=3, txn=1`

**Output:**

| Field | OLD (buggy) | NEW (fixed) |
|---|---|---|
| Top-level impressions | `3` ← only slice 2 | `8` ← cumulative |
| Top-level transactions | `1` ← only slice 2 | `2` ← cumulative |
| byDate rows | both kept (orphan Apr 01) | both kept |

| Idempotent on 3rd identical call | n/a | yes (still 8/2) |

The OLD column reproduces the bug's documented failure ("equal to second call's values, not the sum"). The NEW column shows the cumulative result, consistent and idempotent.

## Summary

- Add input-file-size and per-row progress logging around every `splitCsvFile` call in `audience.statara.publish` (activated CSV, gzip split-file, and standard pipelines).
- Pure observability change — no behavior or control-flow edits.
- Lets us tell at a glance whether a publish job is starved of input, hung inside `csv-parser`, or just legitimately grinding through a large CSV.

## Motivation

When `audience.statara.publish` job #859 (and several before it) appeared stuck on partition 1/51, the only log line we had was `[1/51] Splitting into chunks...` followed by 30+ identical `Upload progress: 0 chunks uploaded so far. Pool status: 0/0 connections active` lines from the 15s `setInterval` heartbeat. There was no way to tell whether:

- The downloaded temp file was empty (silent download failure)
- `csv-parser` was hung on a malformed row
- The split was simply taking minutes per partition due to file size

`splitCsvFile` already accepts an `onProgress` callback (`apps/external-service/src/features/publish/utils/split-csv-file.ts`) but none of the three call sites in the statara publish queue were passing one. This PR wires it up and adds a `fs.statSync` size log so the operator can distinguish those cases in the Bull dashboard.

## What changed

In `apps/external-service/src/features/publish/sources/statara/statara-publish-audience.queue.ts` at the three `splitCsvFile` call sites:

1. **Activated CSV pipeline** (`isActivatedCsv` branch) — line ~197
2. **Gzip split-file pipeline** (`isSplitFilePipeline && Key.endsWith('.csv.gz')`) — line ~391
3. **Standard pipeline** (`else` branch's per-file loop) — line ~568

Each call site now emits:

- A pre-split size log: `Downloaded/Decompressed/File N/M ... (X.XX MB). Starting split into chunks of <audiencesSize> rows...`
  - A `0.00 MB` reading is the smoking gun for a silently-failed download/decompress.
- An `onProgress` heartbeat passed to `splitCsvFile`:
  - First row always logs immediately (so we know within seconds if `csv-parser` ever produced a row).
  - Subsequent heartbeats throttled to one log per ≥10 seconds, including rows processed, current chunk index, elapsed seconds, and rows/sec.
- A post-split duration log: `Created N chunk(s) in Xs` / `split into N chunk(s) in Xs.`

Example of the new output in Bull Dashboard for a healthy partition:

```
[Job #859] [1/51] Downloaded ...-statara-part013.csv → /tmp/...-statara-part013.csv (842.31 MB). Starting split into chunks of 3000000 rows...
[Job #859] [1/51] Splitting progress: 1 rows processed → chunk #1 (0.0s elapsed, ~0 rows/s)
[Job #859] [1/51] Splitting progress: 1,800,000 rows processed → chunk #1 (10.2s elapsed, ~176,470 rows/s)
[Job #859] [1/51] Splitting progress: 3,600,000 rows processed → chunk #2 (20.5s elapsed, ~175,609 rows/s)
...
[Job #859] [1/51] Created 9 chunk(s) in 173.4s
```

|---|---|
| `Downloaded ...-partXXX.csv → ... (0.00 MB)` | Download silently failed — investigate `getCsvToTempFile` / S3 access |

| `Splitting progress: 1 rows processed → chunk #1` fires but no further heartbeats for >60s | Disk write is slow or stalled; check pod `/tmp` free space and write throughput |
| Heartbeats firing at ~150k+ rows/s | Healthy — it's just a big file. Wait it out, kill, or follow up with a perf change |

# Ticket Issue #7421 :  

## Fix $0 Spend on Auction Plus Campaigns

## Issue

Campaigns with `pricingModel = "Percentage"` (Auction Plus) showed $0 spend on the Campaign Details > Summary tab, even though delivery and DSP-side cost data were correct.

Verified for campaign `cmnf3rtzqu6980b16o8xpm27k`:
- S3 display report has `sumTotalCost: 182.43` across 37/38 dates ✓
- Campaign aggregate `aggTotalCost: 182.42` ✓
- Dashboard Summary showed `Spend: 0` ✗

CPM campaigns have the same bug but the per-day amount is small enough nobody flagged it.

## Root cause

In `combineAllByDateData` (consolidate.ts), the merge initialized `spend: 0` from the display side, then only restored it when a matching transaction byDate row existed for that date.

```js
// BEFORE
spend: 0,                              // wiped on init
existing.spend = entry.spend || 0;     // only restored if txn exists on this date
```

Auction Plus campaigns have many delivery days but few transaction days → most dates end up `spend: 0` regardless of the display-side cost.

## Fix

Pull `spend` from the display entry on init, and protect it during the transactions merge.

```diff
- spend: 0,
+ spend: entry.spend || 0,
- existing.spend = entry.spend || 0;
+ existing.spend = entry.spend || existing.spend || 0;
```

Applied in both versions of the job:
- `src/jobs/consolidate-report-s3/utils/consolidate.ts` (manual local invocation)
- `src/jobs/consolidate-report-s3-prisma6/utils/consolidate.ts` (Airflow production)

## No regression

When transactions byDate has a spend value, it still wins — and that value is itself sourced from `displayByDate.spend` (in `consolidateTransactionWithEvents`), so the number is the same either way. The fix only changes behavior for dates that have delivery but no attributed transactions.

# Ticket Issue #11318 :

# Server-side sort for Campaign Details → Transactions table

## Problem

In the Campaign Details view's Transactions tab, clicking a column header (Order, Last Impression, Impressions, Ordered on, Total) only sorted the rows on the current page. Because the table is server-side paginated, the other pages were never re-ordered — so "sort by Total descending" actually returned the top 10 of whatever page you were on, not the top 10 overall.

Root cause: the Ant Design table had client-side `sorter` comparator functions on each column, the `onChange` handler ignored the `sorterObject`, and `TransactionsTabLoader` passed a hardcoded `orderBy: 'transactionDate_ASC'` to the GraphQL query.

## Solution

Frontend-only fix — the backend already supports `orderBy` on `campaignOrderTransactionsWithEventsesConnection` (Prisma v1 auto-generates `<field>_ASC` / `<field>_DESC` for every scalar on `CampaignOrderTransactionsWithEvents`), so no `mediajel-gql-service` change was needed.

- `TransactionsTable.js`
  - Replaced each column's client-side comparator with `sorter: true` and a controlled `sortOrder` driven by URL params.
  - New `onChange` handler captures the Ant Design `sorterObject` and persists `sortField` / `sortOrder` to the URL (and resets to page 1 on a sort change).
- `TransactionsTabLoader.js`
  - Reads `sortField` / `sortOrder` from URL params, validates `sortField` against the allowed scalars, and builds an `orderBy` string.
  - Threads the dynamic `orderBy` into both `CAMPAIGN_ORDER_TRANSACTIONS` (paginated) and `CAMPAIGN_ORDER_TRANSACTIONS_ALL` (lazy export query). Default remains `transactionDate_ASC` when no user sort is active.

## 📝 Why bundled rather than split

Each section below describes one logically separable change. We chose to ship them together because:

1. **All six branches were active simultaneously during incident triage.** Splitting into four PRs now would mean rebasing four diverging branches against a still-evolving master, and the operational signal (a single deploy that stops the incident) would be diluted across four merge windows.
2. **The publish queue, the SFTP pool, and the cancellation endpoints share too much state to land independently.** Part 2's resume cursor depends on Part 3's per-job tempfile naming; Part 5's cancellation depends on the same `processPartitionsStreaming` checkpoints Part 2 introduced; Part 6's SFTP-pool changes are exercised by Parts 2 and 5 alike. 3. **The taxonomy migration (Part 4) is the only fully-independent piece.** It is small enough that the migration-and-feature-flag overhead of a separate PR exceeded the maintenance overhead of bundling here.

We're documenting the scope honestly rather than trying to make a four-PR sized change look smaller. The [Test plan](#-test-plan) at the bottom is structured per-part so the reviewer can verify each piece in isolation.

## 1️⃣ Storage-leak fix on the source CSV (Part 1)

Before this PR, the downloaded source CSV in `processPartitionsStreaming` was unlinked only on the happy path; if anything between download and unlink threw (most realistically `splitCsvFile`), the multi-GB temp file leaked. Bull's default 5 attempts compounded the leak per job. Both code paths (plain activated-CSV and gzip split-file) had the same shape and the same gap.

Fix: wrap each download + split + unlink sequence in `try { ... } finally { if (fs.existsSync(path)) fs.unlinkSync(path); }`. Per-chunk deletion (already correct in master via PR #2167) is untouched.

## 2️⃣ Resume from last confirmed chunk (Part 2)

A new `PublishResumeState` helper class persists `{ donePartitions, inProgress }` into `job.data.resume` via Bull's `job.update()` — Redis-backed, survives both Bull retry attempts and pod restarts via Bull's stalled-job recovery. `splitCsvFileStreaming` gains two optional params (`skipUntilGlobalChunkIndex`, `globalChunkIndexOffset`) so it discards rows for chunks already confirmed on a prior attempt. `processPartitionsStreaming` consults the helper at partition boundaries (skip already-confirmed partitions) and at chunk boundaries (mark confirmations only after SFTP upload AND S3 backup both succeed).

**Architectural decisions** (full design at `docs/superpowers/specs/2026-05-15-statara-publish-resume-design.md`):

| Decision | Choice |
|---|---|
| Checkpoint granularity | Hybrid: per-partition done-flag + per-partition chunk cursor for the currently-in-progress partition |
| Storage | Bull `job.data` via `job.update()` (Redis), with the partition-done marker now also persisted to Redis under its own key for cross-attempt durability (see Part 6 amendment) |
| "Done" criterion for a chunk | SFTP upload AND S3 backup both succeeded (reducing to SFTP-only when `hashedEmailListId` is null) |
| Pipeline scope | Both Nexxen pipelines (activated-CSV + gzip split-file) via the shared `processPartitionsStreaming` helper |

**Load-bearing operational invariant:** Nexxen SFTP must tolerate same-filename overwrite. On the boundary chunk (worker crashed between SFTP upload and `markChunkConfirmed`) the resume re-uploads under the same filename. **This needs a smoke test in staging after deploy.**

## 3️⃣ Per-job tempdir-sweep scoping (Part 3)

Discovered during code review of Part 2: with `replicas: 2` for external-service and `@Process({ concurrency: 1 })` per pod, up to TWO statara publish jobs run concurrently. The existing tempdir sweep (`cleanupTempStataraArtifacts` from PR #2167) does a glob-match against `*statara*.csv` and unlinks files older than 60s by mtime — but `mtime` doesn't refresh on reads, so a multi-GB job whose source CSV download finished 65s ago has a "stale" mtime even though it's still actively being read/split. The other pod's sweep deletes it. The real correctness hazard is chunk files mid-batch: a chunk created at T=0, queued behind SFTP pool pressure until T=70s, would be eligible for deletion in that window; the worker then calls `sftpPool.uploadLocalFile(chunkPath, …)` and gets `ENOENT`.

Fix: embed Bull `job.id` in source-file basenames (`<tmpdir>/job-${id}-<timestamp>_<basename>`); scope the sweep's match rule to require `job-${jobIdScope}-` in the basename. Chunks inherit the prefix as siblings. `STALE_MTIME_MS` drops from 60s to 10s since per-job scoping replaces it as the primary safety mechanism.

## 4️⃣ Taxonomy persistence across activate/publish retries (new in this PR)

Before this PR, `ensureNexxenTaxonomy` minted a new Nexxen taxonomy id every time `publishToNexxen` ran — including on Bull retries of the same publish job and on manual operator re-triggers. That created orphan taxonomies in Nexxen's drop and surfaced as "duplicate audience" alerts. Fix: a new nullable `Audience.taxonomyId` column persists the minted id on first success; subsequent attempts find the existing id and reuse it instead of regenerating.

The migration is nullable + additive so it's safe to deploy without coordinating downstream schema reads. The generated Prisma DTOs are picked up automatically by `prisma generate`.

## 5️⃣ Cooperative-cancel endpoints (new in this PR)

Two new endpoints let operators terminate stuck publish jobs without restarting pods:

- `POST /audiences/statara/activate/clear-jobs[?jobId=...]` (internal-service)
- `POST /publish/nexxen/clear-jobs[?jobId=...]` (external-service)

**Cancellation semantics (revised after review):** the original implementation force-deleted Bull's `:active` and `:lock` keys directly via Redis to make the job disappear from the dashboard immediately. The reviewer flagged that this races with the in-flight worker: under `replicas: 2`, another pod could acquire the just-freed job id while the original was still mid-`client.put`, double-uploading under the same filename. The current implementation signals a cooperative cancellation through a distributed Redis flag (`cancel:<queueName>:<jobId>`, 1h TTL), workers on every pod observe the flag at their existing checkpoints (chunk loop, retry backoff, SFTP slot acquisition), the HTTP handler waits up to 45s for the lock to release through Bull's normal lifecycle, and only then removes the job via the standard `job.remove()` API. Workers that don't settle in time stay in place; the cancellation continues in the background and the next checkpoint or Bull retry cycle finishes the cleanup.

This is slower than the original force-delete (HTTP latency up to 45s when an active job is in flight) but eliminates the double-upload race. Operators see a `"cancellation in flight"` message rather than `"job cleared"` when a worker is still inside `client.put` and the watchdog hasn't fired yet.

## 6️⃣ SFTP-pool hardening and stall watchdog (new in this PR)

Real-world failure modes that surfaced during the incident drove a set of pool-level changes:

- **`PUT_TIMEOUT_MS` watchdog** wraps every `client.put` so a wedged ssh2 stream (keepalives still flowing but the write neither resolves nor rejects) is force-rejected after 30 minutes. **New in the post-review revision:** tunable via `NEXXEN_SFTP_PUT_TIMEOUT_MS` env so staging can exercise stall paths quickly.
- **Unhealthy-connection recycling** marks a pool slot as bad when a put rejects with `ECONNRESET` / `ETIMEDOUT` / `Unexpected end event` / put-timeout, so the next caller picks a fresh connection instead of reusing the dead one.
- **`listRemoteDirectory()` / `deleteRemoteFile()` / `shutdownAll()`** support the SFTP-resume index (Part 2), the partial-chunk cleanup before re-upload (chunk retry classifier), and the stall watchdog's forced pool teardown.
- **Stall watchdog**: a 15-minute zero-progress timer at the publish-queue level aborts the job and tears down the pool when no chunk progress is observed. **New in the post-review revision:** tunable via `STATARA_STALL_THRESHOLD_MS`.
- **Partition-completion cursor moved off SFTP** (post-review revision): the original implementation wrote `.done` marker files to the Nexxen drop directory. Nexxen's ingestor expects only CSVs there. Markers are now persisted to Redis under `statara:publish:partition-done:<strippedMarker>` with a 14-day TTL. A backwards-compatible read path still recognizes legacy `.done` files so jobs that were mid-publish at deploy time continue to skip already-completed partitions; new markers are Redis-only.

## ⚡ Performance impact

Positive in failure scenarios:
- No more re-uploading already-confirmed chunks on retry (Part 2)
- No more re-downloading source CSVs for already-confirmed partitions (Part 2)
- No more lost partitions from cross-job sweep race (Part 3)
- Taxonomy reuse eliminates redundant CSV-rewriting on the Nexxen taxonomy file on every retry (Part 4)

Neutral on the happy path:
- One additional `job.update()` per chunk (small Redis write) — Part 2
- One Redis read per partition at job start to populate the partition-done Set — Part 6 amendment
- A 3-second Redis poll per active worker for the cooperative cancel flag — Part 5

Slightly negative:
- `clear-jobs` HTTP latency goes from ~50ms to up to 45 seconds when an active job is in flight. This was the explicit tradeoff for B1 race-safety; the operator alternative (force-delete) was unsafe under `replicas: 2`.

## 💥 Breaking changes

None functional. Behavior changes that operators should be aware of:
- `clear-jobs` no longer instantly removes active jobs from the Bull dashboard; the dashboard shows the job as `active` until the worker reaches its next checkpoint and releases the lock. This is the price of B1 race-safety.
- Partition-completion markers are no longer visible in the Nexxen SFTP drop directory. Anything that monitored that directory for `.done` files should switch to the `statara:publish:partition-done:*` Redis keys.

Worst case is wasted CPU re-splitting chunks the SFTP-resume-by-size layer then no-ops on — not duplicate data. See the runbook comment at `statara-activate.queue.ts` for full detail.
2. After deploy, run through the test plan above on staging.
3. Monitor for chunk-upload `ENOENT` errors in the week after deploy; with this fix they should disappear.

**Repo:** [MediaJel/mediajel-gql-service](https://github.com/MediaJel/mediajel-gql-service)
**Merged:** 2026-05-01T18:38:51Z
**PR:** [#7387](https://github.com/MediaJel/mediajel-gql-service/pull/7387)

## Summary

Mirrors the existing `CampaignToolsQueue.reprocessByCadence` pattern for **editorials** so we can reprocess an editorial's lifetime in fixed-size windows instead of one giant pass.

The orchestrator walks an editorial's lifetime in N-day windows and enqueues a per-window reprocess job. Each window runs **transactions + consolidate**; only the **final `lastRun` window** also runs **sessions**, which is the heavy publisher-attribution lookup (analogous to `noAttributionLookup: !lastRun` on the campaign-tools side).

This keeps non-final windows cheap and avoids re-running attribution every step, while still producing a fresh, attribution-correct result on the last window.

## What's in the change

- **New cadence orchestrator** in both editorial flavors:
  - [src/jobs/editorial-prisma6/queues/cadence.queue.ts](src/jobs/editorial-prisma6/queues/cadence.queue.ts) + [process/cadence-editorial.process.ts](src/jobs/editorial-prisma6/process/cadence-editorial.process.ts)
  - [src/jobs/editorial/queues/cadence.queue.ts](src/jobs/editorial/queues/cadence.queue.ts) + [process/cadence-editorial.process.ts](src/jobs/editorial/process/cadence-editorial.process.ts) (prisma1 mirror)
- **Window override on existing processes:** `editorial.new.process.ts` and `editorial-sessions.process.ts` now honor an optional `reprocessConfig.{startDate,endDate}` so a single window can be processed in isolation. Default behavior is unchanged when no config is passed.
- **Resilience:** Wraps the four `prismaServer` calls in `editorial-sessions.process.ts` (prisma1) with `retryWithBackoff` to recover from transient `graphql-tools` "fetch failed" errors that have been showing up during long sessions runs.
- **Entry points:** New `function=cadence` branch added to `src/jobs/editorial/index.ts` and `src/jobs/editorial-prisma6/index.ts`, plus matching shared types in `utils/interface.ts` / `utils/args.ts`.
- **npm scripts:**
  - `dojo:local-editorial-cadence`
  - `prod:local-editorial-cadence`

## How to run

```bash
# Dojo
npm run dojo:local-editorial-cadence -- args id=<editorialId> cadence=<days>

# Production
npm run prod:local-editorial-cadence -- args id=<editorialId> cadence=<days>
```

`cadence` is the window size in days (e.g., `cadence=7` → 7-day windows). Editorials don't have a real `endDate`, so the walk always clamps to today and the final window is flagged `lastRun: true`.

### [gql-service] Fix Paid Search Missing Dates (#7504)

**Repo:** [MediaJel/mediajel-gql-service](https://github.com/MediaJel/mediajel-gql-service)
**Merged:** 2026-05-21T06:09:30Z
**PR:** [#7504](https://github.com/MediaJel/mediajel-gql-service/pull/7504)

# Description

Paid Search reports were stuck at **2026-05-07** for affected orgs. Five compounding issues in the daily-metrics save path were preventing new dates from persisting.

## Root causes

1. **Stale `PaidSearchDateUnit` rows held the globally `@unique` `key`.** `setByDatesReport` / `setCampaignByDatesReport` build the upsert key as `${parentId}_${xAxis}` and run a nested upsert. When rows with these keys existed but were linked to a *different* `paidSearchUnit` (orphan or replaced relation), the nested `where: { key }` could not reach them, Prisma fell through to CREATE, and the global `@unique` rolled back the entire 792-row mutation. Nothing has saved since the orphans appeared. Fixed by purging matching keys before the upsert (`deleteManyPaidSearchDateUnits` for Prisma v1, `prisma6.paidSearchDateUnit.deleteMany` for Prisma 6).

2. **Campaigns with no end date pulled ~14 years of segmented data.** Google Ads returns `2037-12-30` as the "no end date" sentinel. `getCampaignSegmentedByDateById` passed it through unchanged, causing the `external-service.dmp.cnna.io/api/connections/google/ads/searchStream` call to ETIMEDOUT. Clamped `cmp.endDate` to `min(cmp.endDate, today)` at the call site.

3. **`googleAdsQuery` returning `null` crashed `segmentedRecordBuilder`.** After the timeout exhausted retries, `null` was passed to `for (const d of data)`, throwing and killing the whole `getCampaigns` run. Added a guard so `getCampaignSegmentedByDateById` returns `[]`.

4. **Chunk slice off-by-one.** `slice(i, i + 999)` with `i += 1000` silently dropped 1 record per chunk above 1000 entries. Fixed to `slice(i, i + 1000)` in `setByDatesReport`, `setCampaignByDatesReport`, and `setAggregatedByDate`.

5. **`concat(false)` polluted the consolidated byDate.** When `hasGA = false`, `byDateMetrics.concat(keywords, locations, !isNil(campaignGA) && campaignGA)` appended the literal `false`, which `getByDate` turned into a junk row with `date: undefined`. Replaced with explicit spreads + `Array.isArray(campaignGA) ? campaignGA : []`.

All five changes are mirrored across `src/jobs/paid-search/` (Prisma v1) and `src/jobs/paid-search-prisma6/` (Prisma 6).

### [microservices] fix(data-factory): stop ingestion scheduler infinite loop (#2232)

**Repo:** [MediaJel/amplication-nestjs-microservices](https://github.com/MediaJel/amplication-nestjs-microservices)
**Merged:** 2026-05-21T20:12:59Z
**PR:** [#2232](https://github.com/MediaJel/amplication-nestjs-microservices/pull/2232)

## Summary

Two interacting defects from #5330aeb2 caused `DataIngestionConsumer` to re-arm `scheduled_config_<id>_*` jobs every ~6ms after each completion, hammering external DSP APIs (e.g. Jane) with real credentials.

- **Defect A** — `ensureScheduledJobExists` only inspected `getJobs(['delayed'])`. Jobs added with `delay: 0` (or whose delay had elapsed) live in `waiting` or `active`, never `delayed`, so the safety net always concluded "no scheduled job exists" and re-queued another. **Fix:** widened the filter to `['delayed','waiting','active','paused']`.
- **Defect B** — `calculateDelayUntilNextSync` legitimately returns `0` when `lastSyncDate` is null (first run) or stale (caught-up incremental syncs, target-end-reached early returns that don't advance `lastSyncDate`). `scheduleNextIngestionForConfig` passed that `0` straight to `queue.add({ delay: 0 })`. **Fix:** clamp to a full `getDelayFromSyncFrequency(...)` interval when raw delay is 0 — a cron-style "Every X" schedule should never fire "now".

### Observed symptom (before fix)

Logs spewed three lines every few milliseconds for config `cmpd3f3x70001tnevekvobrk0` while pretending the cadence was "Every 6 hours":
```
[02:15:55.611] Scheduled next ingestion for config X (Every 6 hours) at 2026-05-21T18:15:55.611Z
[02:15:55.611] Safety net scheduled next ingestion for config X at 2026-05-21T18:15:55.611Z
[02:15:55.612] Starting ingestion job execution at 2026-05-21T18:15:55.612Z
```
50 jobs piled into the queue's COMPLETED tab in seconds while the active job sat at 20% — the consumer was finishing jobs faster than Bull Board could refresh.

### Why "Every 6 hours" logged as a ms-cadence

The "Every 6 hours" string is just a description from `getSyncFrequencyDescription(config.syncFrequency)`. The logged `nextRunTime = new Date(Date.now() + delay)` was "now" because `delay === 0`. Description and computed delay were decoupled.

### Blast radius this fix removes

- External API spam: hammering Jane's `GET /stores/{storeId}/reservations` at hundreds of req/sec with a production apiKey.
- DB write storm in `DataFactoryConfigIngestionJob`.
- ClickHouse `SELECT COUNT(*)` flood from per-job verification.
- Redis/Bull churn and log volume.

No data corruption (inserts were no-ops via dedup), but everything else was on fire.

- `apps/internal-service/src/features/data-factory/queues/data-ingestion.queue.spec.ts` — regression test for `waiting`-state safety-net dedup + updated state-filter assertions

### [gql-service] Fix paid-search-prisma6 crashes: empty campaigns and Decimal aggregation (#7516)

**Repo:** [MediaJel/mediajel-gql-service](https://github.com/MediaJel/mediajel-gql-service)
**Merged:** 2026-05-22T18:12:21Z
**PR:** [#7516](https://github.com/MediaJel/mediajel-gql-service/pull/7516)

No linked issue — production hotfix for `paid-search-prisma6`.

## Summary

The `paid-search-prisma6` job was failing in production on two unrelated bugs introduced during the Prisma 1 → Prisma 6 migration. Both surfaced while processing Google paid-search data.

### 1. Crash on campaigns linked to a different summary

Error: `TypeError: Cannot read properties of undefined (reading 'id')`

`setPaidSearchCampaign` looks campaigns up with a global `findUnique` on `adProviderId`. When a matched campaign belonged to a different org's `paidSearchSummary`, the `update` branch never re-linked it to the current org's summary, so the return query came back with an empty `paidSearchCampaigns` array. `google.process.ts` then dereferenced `paidSearchCampaigns[0].id` and crashed.

- `queries/set-prisma.ts` — the `update` branch now reconnects the campaign to the current org's summary (`PaidSearchSummary: { connect: { id: summaryId } }`), matching the Prisma 1 nested-upsert behaviour.
- `process/google.process.ts` — guards `paidSearchCampaigns[0]` and skips the campaign with a warning instead of crashing.

### 2. Aggregated channel totals concatenated instead of summed

Error: `Invalid value for argument 'cost': invalid digit found in string. Expected decimal String.`

Prisma 6 returns `Decimal` columns (`cost`, `spend`, `revenue`, `conversions`) as Decimal objects. `aggregateProcess` summed the Google and Bing channel totals with the bare `+` operator, which concatenates the Decimal string forms instead of adding them (e.g. `cost` became `"72802.418977999891518.83"`). `setAggregated` then passed that malformed string to `paidSearchAggregatedUnit.create`, which rejected it.

- `process/aggregated.process.ts` — coerces every channel operand with `Number(x) || 0` before summing, matching the existing `getAggregatedByDate` convention. `Int` columns (`clicks`, `impressions`) were unaffected but wrapped for consistency.
